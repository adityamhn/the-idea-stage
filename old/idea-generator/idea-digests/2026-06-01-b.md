## 1. Supply-chain security for the agent toolchain (MCP tool-poisoning scanner + trusted registry)

**Problem.** A team wiring an autonomous agent to a dozen MCP servers and third-party tools has no way to vet what those tools actually do *before and during* use. Tool descriptors are metadata the model reads but humans never review, so a trojanized or typosquatted MCP server can inject instructions, exfiltrate data, or silently change behavior on update. This is a per-install, per-update risk and the blast radius is the agent's full delegated privilege.

**Why unsolved today.** Existing AppSec/SCA tools (Snyk-style) scan code dependencies, not *tool descriptors, schemas, and runtime tool behavior*. The MCP ecosystem shipped with weak defaults: a security analysis documented 30+ CVEs targeting MCP between January–February 2026, with 82% of implementations vulnerable to path traversal and 38–41% lacking authentication; Anthropic's own MCP had a design flaw enabling RCE, and Microsoft's MCP server (CVE-2026-26118) allowed AI tool hijacking. Workaround today is "trust the README and hope," or block third-party MCP entirely.

**Why 2028-2029.** *Signal:* tool poisoning is already named the highest-leverage attack on enterprise agents in 2026; CVEs are accumulating fast. *Speculation:* as agents pull from open MCP marketplaces at npm-like scale into 2028-2029, "is this tool safe to connect?" becomes a gate in every agent deployment — the SCA/Dependabot moment for the agent toolchain. The bet: the supply-chain surface grows faster than the platform vendors patch defaults, leaving room for a dedicated scanner + signed-registry layer.

**Funding / market signal.** Adjacent NHI/agent-security capital is heavy (Oasis $120M Series B; GitGuardian $50M; Noma $100M from the prior run's tally), and the MCP-specific CVE wave is documented. ([MCP 30 CVEs in 60 days](https://agent-wars.com/news/2026-03-13-mcp-security-2026-30-cves-in-60-days-what-went-wrong), [Anthropic MCP RCE / supply chain](https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html), [CVE-2026-26118 Microsoft MCP](https://www.pointguardai.com/ai-security-incidents/microsoft-mcp-server-vulnerability-opens-door-to-ai-tool-hijacking-cve-2026-26118), [MCP tool poisoning](https://itecsonline.com/post/mcp-tool-poisoning-enterprise-ai-agent-security-2026))

**How it makes money.** OSS scanner CLI/CI action (lands the dev champion, mirrors Pentest Copilot OSS→enterprise) → enterprise with a private signed registry, continuous descriptor monitoring, and policy gates. Usage-based per tool/server scanned + seat tier for AppSec. Founder-led to AppSec/platform-security buyers — Aditya's network.

**My edge.** Direct collision of offensive-security depth + agent-internals depth (tool registries, MCP, schema handling). He knows both how tool descriptors are parsed and how an attacker abuses metadata — an uncommon pairing, and exactly the analysis this product performs.

**Strongest counterargument.** Likely a *feature* of the platform vendors, not a standalone company: Anthropic/Microsoft/OpenAI will harden defaults and ship first-party registries with signing, and buyers prefer the platform's built-in trust list over a third-party scanner. The CVE wave may be an early-protocol-immaturity blip that closes as the ecosystem standardizes, shrinking the durable surface.

**Scores.** unsolvedness 4 (tool-descriptor scanning barely exists) · timing 4 (CVE wave now, marketplace scale later) · funding momentum 3 (adjacent capital heavy, this exact wedge unfunded) · monetization clarity 4 (proven SCA/security model) · founder fit 5 (offensive sec + MCP internals) · defensibility 3 (platform vendors can absorb).

## 2. Behavioral-compatibility regression gating for model upgrades & deprecations (the "semver lie" safety net)

**Problem.** Every time a provider bumps a model version, deprecates an endpoint, or an agent framework changes how it emits tool schemas, a team's production agent can silently break or drift — same API, different output distribution. The engineer finds out from a customer, not a test. This happens on a forced schedule (deprecation windows) and ad hoc (minor bumps), and the failure is silent until it's expensive.

**Why unsolved today.** Provider deprecation policies guarantee *availability* ("you can call this endpoint for N months"), never *behavioral compatibility* ("v-next answers your prompts the same way"). Observability tools detect drift *after* it ships; eval tools measure quality in dev but aren't wired as a *pre-swap diff* across model versions and providers. The canonical 2026 case: an n8n v2.4.7→v2.6.3 bump changed tool-schema generation, breaking OpenAI and Anthropic function calls simultaneously — and the same pattern hit FlowiseAI, Zed, and the OpenAI Agents SDK. Workaround: pin versions and pray, then scramble when forced to upgrade.

**Why 2028-2029.** *Signal:* the average enterprise already evaluates or operates ~7 models; IDC projects 70% of leading AI enterprises on multi-model routing by 2028. *Speculation:* with that many models and forced deprecation cadences, "prove the new model is behaviorally safe for my agents before I switch" becomes a standing CI gate by 2028-2029 — a regression suite specifically for model swaps and provider migrations. The bet rides on model churn accelerating, not slowing.

**Funding / market signal.** Routing/gateway infrastructure is hot: OpenRouter closed a $113M Series B (May 26, 2026, CapitalG, ~$1.3B valuation), routing across 400+ models; multi-model routing is becoming standard. ([OpenRouter $113M Series B](https://www.techtimes.com/articles/317353/20260529/ai-gateway-openrouter-raises-113m-google-nvidia-route-between-their-models.htm), [the semver lie: minor LLM updates break production](https://tianpan.co/blog/2026-04-29-semver-lie-llm-minor-update-breaks-production), [n8n schema-drift incident / silent degradation](https://www.traceloop.com/blog/catching-silent-llm-degradation-how-an-llm-reliability-platform-addresses-model-and-data-drift))

**How it makes money.** Usage-based per model-version diff run + seat tier for the platform team; OSS diff/replay harness → enterprise with the historical baseline store, provider-migration playbooks, and CI gating. Sold to AI-platform/infra teams.

**My edge.** Squarely his agent-reliability-infra lane plus his specific multi-provider routing/gateway experience (OpenAI/Anthropic/Groq/Gemini/xAI/OpenRouter) and self-correcting schema-repair SDKs — the n8n failure is literally a schema-repair-plus-regression problem he has already built primitives for.

**Strongest counterargument.** The gateways own the swap point and the traffic: OpenRouter ($113M) or the eval/observability incumbents can add "regression check before you switch models" as a native feature, and they already see every request. It may also be too narrow — a capability inside a broader eval/observability suite rather than a company — and adoption depends on teams treating model swaps as releases, which many still don't.

**Scores.** unsolvedness 4 (pre-swap behavioral diff is genuinely missing) · timing 4 (multi-model now, churn rising) · funding momentum 3 (gateway capital heavy, this exact wedge open) · monetization clarity 3 (narrow; risk of being a feature) · founder fit 5 (his routing + schema-repair edge exactly) · defensibility 3 (gateways/observability adjacent).

## 3. Runtime budget enforcement & circuit-breakers for autonomous agents (agent FinOps that actually *blocks*)

**Problem.** An autonomous agent stuck in a loop or handed an open-ended task can burn thousands of inference calls with no ceiling — and finance finds out on the invoice. Teams have *visibility* dashboards but not *enforcement*: nothing halts a runaway session in real time. This happens per-session, at scale across an agent fleet, and the loss is direct cash.

**Why unsolved today.** Traditional FinOps alerting is built for infra cost that moves over hours/days; it lacks the time resolution to catch a runaway agent session that burns budget in minutes. Cost-visibility tools (Vantage-style) measure spend; they don't sit inline to *block* before overspend. Only 44% of orgs have any AI financial guardrails (Gartner, Mar 2026), yet 98% now manage AI spend (FinOps Foundation 2026) — a wide visibility-vs-enforcement gap. Workaround: post-hoc invoice shock, or crude global API key limits that kill good and bad traffic alike.

**Why 2028-2029.** *Signal:* a documented ~$400M collective cloud-spend leak across the Fortune 500 from agent sessions without per-session ceilings; a public $47k runaway-loop incident; agents burn ~50x the tokens of chat. *Speculation:* as agent fleets become the default workload into 2028-2029, per-task inference budgets and circuit-breakers become a non-negotiable control plane line item — unit-economics enforcement, not just reporting. The bet: spend pain scales with autonomy, and "block before burn" beats "report after burn."

**Funding / market signal.** FinOps for AI named the #1 skillset teams need (FinOps Foundation 2026); Google Cloud shipped AI Spend Caps at Next '26; gateway infra (where enforcement naturally lives) is well-capitalized (OpenRouter $113M Series B). ([$400M AI FinOps enforcement gap](https://www.waxell.ai/blog/ai-agent-finops-cost-enforcement), [Gartner 44% guardrails / 98% manage spend](https://siliconangle.com/2026/05/28/finops-ai-spending-boardroom-strategy-finopsx/), [agents burn 50x tokens / runaway loops](https://leanopstech.com/blog/agentic-ai-cost-runaway-token-budget-2026/), [Google Cloud AI Spend Caps](https://cloud.google.com/blog/topics/cost-management/introducing-spend-caps-ai-cost-visibility-next26?hl=en))

**How it makes money.** Usage/percent-of-savings or seat hybrid; OSS inline budget-enforcer / circuit-breaker proxy → enterprise with fleet policy, per-team accountability, and anomaly auto-halt. Sold to platform/AI-ops and the budget owner.

**My edge.** Multi-provider routing/gateway depth (he's built the layer requests flow through) plus long-horizon-agent failure knowledge — he knows exactly the loop/retry-storm patterns that need a circuit-breaker, and where to intercept them.

**Strongest counterargument.** The enforcement point belongs to whoever owns the gateway, and the cloud/gateway incumbents are already moving in: Google shipped Spend Caps, OpenRouter sits inline with $113M behind it, and enterprises prefer cost controls native to the platform they already route through. Hard to win a standalone "cost" product when the natural home is a feature of infrastructure customers already buy — and "percent of savings" pricing erodes as platforms make basic caps free.

**Scores.** unsolvedness 3 (visibility solved, enforcement partly emerging) · timing 4 (spend pain acute now) · funding momentum 3 (gateway capital heavy; enforcement-as-product thin) · monetization clarity 3 (savings-based pricing fragile) · founder fit 4 (his gateway + loop-failure edge) · defensibility 2 (gateways/clouds own the choke point).

## 4. Prompt-injection defense + continuous injection testing for agentic browsers / web agents

**Problem.** An agentic browser or web agent acting with the user's authenticated privileges executes natural-language instructions hidden in page content, screenshots, or URL params — reaching banks, email, and corporate systems across origins. Every site the agent visits is attack surface, the same-origin policy is moot, and the failure (account takeover, data exfil) is severe. There is no continuous, automated way to test or defend a deployed web agent against this.

**Why unsolved today.** Indirect prompt injection is openly called the unsolved problem inside every agentic browser. 2025 saw real exploits — Brave's indirect-injection disclosures in Perplexity Comet, screenshot injection in Comet/Fellou, Tenable's "Gemini Trifecta," LayerX's one-click Comet hijack via URL params. Platform defenses are nascent (Google's separate "User Alignment Critic" model for Chrome). Defense is point-in-time and per-vendor; there's no goal-directed, continuous injection harness that exercises a web agent across the real adversarial web. Workaround: static guardrail filters and manual purple-teaming.

**Why 2028-2029.** *Signal:* mainstream browsers (Chrome, Edge, Safari) plus Comet and OpenAI's Atlas all shipped agentic features in 2025; arXiv work already demonstrates a "parallel-poisoned web only AI agents can see." *Speculation:* as web agents go mainstream for real transactions into 2028-2029, "did you continuously test your web agent against injection?" becomes a procurement/insurance question the way external pentest is for web apps — and a defense-plus-testing layer becomes standard. Bet leans on his thesis: a *reliable autonomous attacker* exercising a browser agent over many steps is the moat.

**Funding / market signal.** Federal agencies are being told to purple-team AI browsers in 2026; platform vendors are investing in layered browser defenses (Google's User Alignment Critic). Specific *startup* funding for web-agent injection defense was not found in this run — flag as early/under-funded rather than asserting a number. ([agentic browser security year-end review](https://www.wiz.io/blog/agentic-browser-security-2025-year-end-review), [prompt injection unsolved in agentic browsers](https://www.softwareseni.com/why-prompt-injection-is-the-unsolved-problem-inside-every-agentic-browser/), [Google Chrome agentic injection defenses](https://www.securityweek.com/google-fortifies-chrome-agentic-ai-against-indirect-prompt-injection-attacks/), [federal agencies AI-browser purple-teaming](https://fedscoop.com/ai-web-browsers-federal-agencies-purple-teaming/))

**How it makes money.** OSS injection-fuzzing harness for web agents (extends Pentest Copilot's browser-control directly) → enterprise continuous injection-testing service + a defense filter. Usage per agent-target/scan + retainer — the pentest-SaaS economics he ran at BugBase. Sold founder-led to AppSec/security teams shipping web agents.

**My edge.** Strong triple fit: browser-control + long-horizon autonomous execution (the attacker) + offensive-security depth. His existing OSS agentic-pentest framework and browser-control primitives point straight at this; very few can build the goal-directed browser attacker.

**Strongest counterargument.** The platform owns the browser and the defense: Google/OpenAI/Perplexity will bake injection defenses (alignment critics, sandboxed agent modes) into the browser itself, potentially neutralizing third-party defense. And it risks overlapping his own prior #2 (agent red-teaming) — if he's already building a general agent attacker, "web agents" may be a vertical of that, not a separate company. Timing risk too: if mainstream web-agent *transactions* stall on safety, the buyer urgency slips.

**Scores.** unsolvedness 5 (injection is openly unsolved) · timing 3 (web agents real but adoption gated by this very risk) · funding momentum 2 (under-funded as a wedge; read either way) · monetization clarity 4 (proven pentest model) · founder fit 5 (browser-control + attacker + offsec) · defensibility 3 (platform vendors own the browser).

## 5. Actuarial reliability-scoring data layer for AI-agent insurers (the underwriting feed, not the policy)

**Problem.** AI-agent liability insurers now exist but can't price risk: they have no continuous, technical, actuarial-grade measurement of how reliable a given customer's agent actually is (failure rates, drift, tool-misuse exposure, injection susceptibility). Underwriting today is a questionnaire and a hope. Every new policy and renewal needs this assessment, and mispricing is existential for the carrier.

**Why unsolved today.** The insurance products arrived faster than the risk-measurement infrastructure. AIUC raised a $15M seed (Nat Friedman, Anthropic's Ben Mann) to insure AI agents; Klaimee (YC) sells liability insurance for agents; Armilla launched standalone AI liability at Lloyd's and expanded limits to $25M+ by Jan 2026 — meanwhile Berkshire, Chubb, Travelers are excluding AI from standard policies and ISO is making AI exclusion the CGL default. Carriers are exposed but underwrite on self-reported questionnaires; nobody supplies a continuous technical reliability score as the actuarial input. Workaround: crude exclusions, or premiums set by guesswork.

**Why 2028-2029.** *Signal:* multiple funded AI-insurance carriers in 2025-2026, expanding limits, plus a hardening market where standard policies exclude AI — creating demand for *someone* to quantify the risk. *Speculation:* by 2028-2029, as agent-caused losses generate claims, carriers need a third-party reliability score (a "credit bureau for agent risk") to price and reinsure — and the neutral measurement layer becomes the durable position underneath the policy. Bet: claims materialize and force rigorous, technical underwriting rather than questionnaires.

**Funding / market signal.** AIUC $15M seed; Klaimee (YC 2026); Armilla $25M+ limits at Lloyd's; ISO/major carriers excluding AI from CGL. ([AIUC $15M seed](https://fortune.com/2025/07/23/ai-agent-insurance-startup-aiuc-stealth-15-million-seed-nat-friedman/), [Klaimee — liability insurance for AI agents (YC)](https://www.ycombinator.com/companies/klaimee), [Armilla $25M limits / first AI liability product](https://medium.com/@purdyhouse/the-first-ai-liability-insurance-product-has-25-million-in-coverage-five-exist-worldwide-575a2903b17b))

**How it makes money.** B2B data/API: priced per agent assessed / per policy underwritten, sold to carriers, reinsurers, and brokers as the underwriting and renewal feed; potential to also sell the same score to the insured enterprise as a premium-reducing certification. Founder-led to a *new* buyer (carriers) — a different motion than his usual security buyer.

**My edge.** His agent-evals/observability/benchmarking depth is exactly what produces a defensible reliability score, and his offensive-security background lets the score incorporate adversarial exposure (injection, tool misuse) that pure-quality evals miss. The measurement *is* the product, and measurement is his lane.

**Strongest counterargument.** The carriers building the policies (AIUC, well-funded and AI-native) will build their own underwriting models in-house — it's their core IP and edge, so they may never outsource the score. Insurance is a slow, relationship-driven sale into a small set of carriers, contradicting his "real usage and revenue fast" preference; and it's the *furthest* from his shipped-product comfort zone (he sells to engineers, not actuaries). Distribution into carriers is a network he doesn't have.

**Scores.** unsolvedness 5 (no actuarial agent-risk feed exists) · timing 3 (insurers early; claims not yet at scale) · funding momentum 4 (multiple funded carriers create the buyer) · monetization clarity 3 (real budget but slow, few buyers) · founder fit 3 (his measurement edge, but unfamiliar buyer/motion) · defensibility 4 (a neutral, data-network score compounds if he gets there first).
